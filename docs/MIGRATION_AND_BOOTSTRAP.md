# Database Migration & Bootstrap

## Current State

The database bootstrap uses **two mechanisms** that coexist intentionally:

### 1. SQLAlchemy `create_all` (primary bootstrap)

**File**: `backend/scripts/seed_db.py:112-113`

```python
async with engine.begin() as conn:
    await conn.run_sync(Base.metadata.create_all)
```

This is the **primary schema definition mechanism** used in development and
Docker Compose deployments. All 8 ORM models in `backend/app/models/` are
auto-discovered via `Base.metadata` and their tables are created on every
seed run.

**Behavior**: `create_all` is idempotent — it creates missing tables and
silently skips existing ones. It does NOT alter existing tables or migrate data.

### 2. Alembic Migrations (compatibility layer)

**Files**: `backend/alembic/versions/001_initial.py`, `002_backend_v4_contract.py`

Two migration revisions exist as a **production-target compatibility layer**.
They mirror the ORM model definitions for teams that require a formal migration
chain. In practice:

- **001_initial.py**: No-op (`pass`). Tables are created by `create_all`.
- **002_backend_v4_contract.py**: Conditionally adds v4 ecosystem and
  governance columns to tables that may already exist from `create_all`. Uses
  idempotent helpers (`_add`, `_idx`) that check for existing columns and
  indexes before creating them.

## Startup Order (all deployment modes)

```
1. PostgreSQL container starts (Docker)
2. SQLAlchemy engine connects
3. seed_db.py runs:
   a. Base.metadata.create_all — ensure all 8 tables exist
   b. Upsert default users (admin, analyst)
   c. Upsert ModelVersion record from champion manifest
   d. Upsert Subscriber records from cleaned parquet
   e. Upsert ChurnPrediction records from recommendations parquet
   f. Upsert Recommendation records (full CRM payload)
   g. Upsert ShapExplanation records from SHAP parquet
   h. Commit transaction
4. FastAPI server starts (uvicorn)
5. MLService warms RuntimeConfig (loads champion artifacts)
```

## Production-Safe vs. Development

| Aspect | Current Behavior | Production Recommendation |
|--------|-----------------|--------------------------|
| Schema creation | `create_all` in seed script | Replace with full Alembic autogeneration pipeline |
| Migration chain | 2 no-op/idempotent revisions | Implement proper `upgrade()`/`downgrade()` for all models |
| Data seeding | Upsert logic (idempotent) | Keep as-is for bootstrapping; add migration-based seed data |
| Idempotency | All operations are safe to re-run | Verified safe |

## What is intentionally legacy/compatibility

- Migration 001 is a **no-op placeholder** — its revision ID anchors the chain
  so future autogenerated migrations can reference it as `down_revision`.
- Migration 002 uses **conditional column addition** (`_has_column` check)
  rather than pure ALTER TABLE — this allows it to run safely whether or not
  `create_all` has already created the columns.

## Reproducing Current State Safely

```bash
# Option A: Clean bootstrap (recommended)
docker compose -f docker/docker-compose.yml up postgres -d
cd backend
python scripts/seed_db.py    # creates tables + seeds data

# Option B: Migrations (if you need the chain)
alembic upgrade head          # applies 001 + 002 (no-ops for DDL)
python scripts/seed_db.py     # seeds data (create_all is skipped for existing tables)
```

## Migration Hardening (deferred)

Full migration hardening is tracked as deferred work:

1. **Replace `create_all` with autogenerated migrations** — run
   `alembic revision --autogenerate -m "description"` to capture any schema
   changes, then remove `create_all` from `seed_db.py`.
2. **Add proper downgrade paths** — current migrations have empty `downgrade()`
   (non-destructive by design).
3. **Add migration tests** — verify that `alembic upgrade head` and
   `alembic downgrade -1` work correctly against a fresh PG instance.

**Current schema is stable** — no schema changes are expected before
production deployment, so the `create_all` + migration-compatibility approach
is safe for the current phase.
